The **Workflows** section of the console lets you create container workflows from a YAML definition and watch them run — viewing the execution diagram, per-step status, live logs, and events.

::callout{color="blue"}
**What a workflow is**: a YAML definition (Docker Compose-style) made up of one or more **steps**. Each step runs a container image on a chosen platform. Steps are grouped into **phases** that execute in order; steps within a parallel phase run together.
::

## The Workflows explorer

Workflows live in the **Workflows** section of the left explorer sidebar.

- **Header**: a refresh button re-lists your workflows.
- **Empty state**: when there are none, the section shows "No workflows yet" with a **New workflow** button.
- **New workflow**: available from the empty-state button, the section's right-click menu, and the **Workflow** menu in the menu bar.
- **Workflow node**: shows a colored **status dot** and the workflow name. Expanding it reveals the **Diagram**, **steps**, **logs**, and **events** entries.

## Creating a workflow

Choose **New workflow** and pick a local YAML file (`.yaml` / `.yml`). The console reads the file and uploads its contents to the engine, which validates the definition and pulls the required images.

```yaml
name: hello-world
steps:
  - name: greet
    platform: docker          # docker | podman | singularity | apptainer
    image: hello-world
```

Definition fields:

| Field | Description |
|-------|-------------|
| `name` | Workflow name (required) |
| `tags` | Optional list of tags |
| `steps[]` | One or more steps (required) |

Each step accepts: `name`, `platform` (default `docker`), `mode` (`parallel` or `sequential`, default `sequential`), `image`, `command`, `env`, `bind`, `ports`, and `resources` (`cpu`, `memory`, `gpu`). Singularity/Apptainer steps require a `command` and cannot expose `ports`.

::callout
**Identity & remote source**: the console creates a workflow from an uploaded file. To set a custom identity or import a definition from a URL, use the CLI (`dxflow workflow create --identity <id> <file>`) or the API (`source` parameter).
::

## Inspecting a workflow

Expanding a workflow opens these views as tabs (logs and events open in the secondary pane):

::tabs
  ::tab-item{label="Diagram" icon="i-hugeicons:git-branch"}
  A read-only layout of the workflow grouped by **phase**, with each step shown as a card carrying its status dot.
  ::

  ::tab-item{label="Steps" icon="i-hugeicons:layers-01"}
  Per-step detail: status, phase/index, process id, and exit code.
  ::

  ::tab-item{label="Logs" icon="i-hugeicons:terminal-01"}
  A **live** stream of combined stdout/stderr for the workflow's steps.
  ::

  ::tab-item{label="Events" icon="i-hugeicons:list-view"}
  A timestamped list of workflow events.
  ::
::

## Statuses

Workflow status is one of `created`, `started`, `stopped`, `exited`. Step status is one of `pending`, `running`, `exited`.

## Lifecycle actions

From the console you can **create** workflows and **Prune all** (remove finished workflows, from the **Workflow** menu).

Starting, stopping, and removing individual workflows is done from the CLI or API:

```bash
dxflow workflow start <identity>
dxflow workflow stop <identity>
dxflow workflow remove <identity>
```

See the [Workflow CLI reference](/docs/cli/workflow) and [Workflow API](/docs/api/workflow) for the full set of operations.

## Supported platforms

Steps can target **Docker**, **Podman**, **Singularity**, or **Apptainer**, selected per step via the `platform` field.

::callout{color="amber"}
**Not part of the workflow engine**: there is no visual drag-and-drop pipeline builder, template library, conditional branching, rollback, rolling updates, or compose version history. Workflows are authored as YAML and executed phase-by-phase.
::
