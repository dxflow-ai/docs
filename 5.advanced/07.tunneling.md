dxflow exposes local services over authenticated WebSocket connections using a proxy-bridge architecture with automatic subdomain allocation.

## Architecture

Traffic routes between a central **proxy hub** and one or more **bridge nodes**.

**Proxy (hub)**

- Allocates subdomains to bridges and routes incoming traffic over WebSocket tunnels
- Manages authentication and traffic limits
- Maintains connection pools and routes by the 12-byte identity

**Bridge (node)**

- Connects to a hub to obtain a subdomain and opens a bidirectional WebSocket tunnel
- Routes traffic between the hub and the local service, with automatic reconnection
- Validates that the upstream is a private/loopback/unspecified address

### Identity system

Every proxy-bridge connection uses a unique identity for subdomain allocation and routing:

- **Format**: lowercase string of 8–12 chars — 2 letters + 6–10 alphanumerics (e.g. `ab1234567890`).
- **Storage**: held in a fixed 12-byte array, zero-padded if shorter.
- **Generation**: created automatically during connection.

### Traffic flow

```
User connects with subdomain
    ↓
UserConnection → ProxyPool.UserSide → Broadcast to Bridge → Wait for pairing
    ↓
Bridge receives broadcast
    ↓
Bridge WebSocket → Decode Identity → Create Gateway Connection → ProxyPool.BridgeSide
    ↓
ProxyPool pairs connections → TcpPair transfers data → Update statistics
```

| Component | Behavior |
|-----------|----------|
| **Connection pooling** | 8-second timeout window for pairing connections; unpaired connections are cleaned up; thread-safe via mutex; connections reused to reduce overhead. |
| **Bidirectional transfer** | TCP traffic flows through WebSocket frames. |
| **Traffic statistics** | Real-time tracking of read/write bytes. |
| **Hijack detection** | Incoming connections are analyzed for subdomain requests. |

## Setting up tunnels

### Default hub

`dxflow.ai` is a ready-to-use public hub — no configuration or deployment needed. It is always available, allocates a subdomain automatically per bridge, and imposes no connection or traffic limits from the hub side.

### Your own proxy hub

Running a private hub requires a specific license. [Contact us](/docs#contact-us) for enterprise licensing and private hub deployment, or see [licensing](/docs/advanced/licensing). A private hub gives you full control over authentication, traffic limits, custom domain, and subdomain allocation on dedicated infrastructure.

### Connect bridge nodes

Bridge functionality requires a bridge-enabled license (not included in the general license — see [licensing](/docs/advanced/licensing)).

```bash
# Start bridge connecting to default hub (dxflow.ai)
dxflow boot up --bridge --daemon

# Bridge receives a subdomain from the hub.
# Connection and traffic limits default to unlimited and are enforced per license.
```

### Manual management

::tabs
  ::tab-item{label="Proxy Management" icon="i-hugeicons:black-hole-01"}
  ```bash
  dxflow proxy list             # List active proxies on hub
  dxflow proxy remove <identity> # Remove a specific proxy allocation
  dxflow proxy prune            # Remove unused proxies
  ```

  There is no `proxy create` command — allocations are created automatically when a bridge connects. The proxy CLI supports only `list`, `remove`, and `prune`.
  ::

  ::tab-item{label="Bridge Management" icon="i-hugeicons:activity-02"}
  ```bash
  dxflow bridge list                 # List bridge connections
  dxflow bridge disconnect <identity> # Disconnect bridge temporarily
  dxflow bridge connect <identity>    # Reconnect bridge
  dxflow bridge prune                # Clean up inactive bridges
  ```

## Authentication

| Credential | Details |
|------------|---------|
| **Proxy credentials** | Auto-generated and required for WebSocket connections: username is a 6-character random nonce, password a 24-character random nonce. |
| **Gateway authorization** | Optional. Authorization tokens passed in query parameters prevent unauthorized subdomain allocation. |

## Traffic limits

| Scope | Default |
|-------|---------|
| **Hub (`dxflow.ai`)** | Unlimited connections and traffic; central routing for all users. |
| **Bridge nodes** | Connections and traffic both default to unlimited (`-1`) until a license sets a limit. Bridge functionality requires a license; the general license does not include it. |

## Security

### Upstream validation

The upstream must be a private, loopback, or unspecified IPv4 address; the scheme must be `http` with an explicit port and empty path. This prevents exposing public services beyond the local network.

### TLS

Gateway connections support HTTPS with proper SNI handling for subdomains and certificate validation.

### Hijack protocol

Routing uses the format `<identity|fingerprint>`:

| Element | Description |
|---------|-------------|
| `identity` | 12-byte bridge identifier |
| `fingerprint` | UUID for connection tracking |

## Performance

| Connection type | Characteristics |
|-----------------|-----------------|
| **Direct proxy** | Minimal overhead, low latency, production-ready. |
| **Bridge tunnel** | WebSocket overhead; development/testing scenarios. |

**Traffic statistics** are reported on both sides. Hub metrics cover user traffic (incoming from internet) and bridge traffic, formatted as `(↓ download) (↑ upload)`. Node metrics cover gateway traffic (to/from hub) and upstream traffic (to/from local service), with bandwidth monitored against limits and connection timestamps.

## Troubleshooting

| Error | Cause / Fix |
|-------|-------------|
| **Proxy not available** | Identity doesn't exist on hub — proxy was removed or never created. Check the proxy list. |
| **Proxy not connected** | WebSocket dropped or network interruption. Verify bridge status and connectivity. |
| **Bridge not allowed** | General license lacks the bridge feature. Upgrade to a bridge-enabled license ([guide](/docs/advanced/licensing)). |
| **Traffic limit exceeded** | License-defined traffic limit reached on the bridge. Upgrade for higher limits ([guide](/docs/advanced/licensing)). |

**Debugging steps:**

1. Enable debug logging: `dxflow --log-level=debug boot up --proxy`
2. Monitor connection pools — check pool sizes and timeout messages.
3. Track traffic stats via the API for real-time bandwidth.
4. Test connectivity of upstream services and WebSocket connections.

## API integration

REST APIs for programmatic control.

**Proxy hub:**

```bash
curl http://hub/api/proxy/             # List active proxies
curl -X POST http://hub/api/proxy/     # Create new proxy
curl -X DELETE http://hub/api/proxy/prune/ # Remove inactive proxies
```

**Bridge node:**

```bash
curl http://node/api/bridge/           # List bridge connections
# Reconnect specific bridge (identity in body/query, not the path)
curl -X PUT http://node/api/bridge/connect/ -d '{"identity":"<identity>"}'
curl -X DELETE http://node/api/bridge/prune/ # Clean up inactive bridges
```
