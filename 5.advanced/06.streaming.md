dxflow's engine serves every endpoint in two modes: buffered (the default) and streaming (chunked). The client picks the mode per request; content type and size never trigger it automatically.

::callout{color="blue"}
Streaming is **opt-in**. Request it with the `stream` query parameter or an `Accept: application/stream+json` header.
::

## Selecting the response mode

The engine resolves the mode in this order:

| Priority | Method | Example |
|----------|--------|---------|
| 1st | Query parameter | `?stream=true` |
| 2nd | `Accept` header | `Accept: application/stream+json` |
| 3rd | Default | none → buffered |

**Streaming** uses HTTP chunked transfer encoding, flushes each chunk immediately, and keeps memory at O(1) per chunk — suited to large datasets and real-time updates.

**Buffered** assembles the full response in memory, sends it in one operation with a `Content-Length` header, and allows error recovery before transmission — suited to small or transactional responses.

## When streaming matters

Streaming only changes behavior on endpoints that emit multiple chunks (lists, logs, event streams). Single-entity reads and simple mutations behave identically in either mode. Good candidates:

- Large file / directory listings
- Workflow execution logs (including the live log endpoint)
- Server-sent event endpoints (e.g. workflow signals)

## Enabling streaming

Add `stream=true` to any API request:

```bash
# File listings
curl "http://localhost/api/artifact/?stream=true"

# Workflow logs
curl "http://localhost/api/workflow/logs/?stream=true"
```

The web console requests streaming automatically for live, list-heavy views: file-manager browsing/search/bulk-ops/archives, workflow live logs and status/metrics, and system status lists (bridges, proxies, shell sessions, health checks).

CLI commands that stream:

```bash
dxflow artifact list /large-directory     # large directory listings
dxflow workflow logs my-workflow --live   # live log endpoint
dxflow ping                               # connectivity testing
dxflow bridge list                        # bridge connections
```

To compare or fall back, force buffered mode with `?stream=false`.

## Response format

Both modes return the same JSON array of typed chunks. A **status** chunk is always first; an optional **total** chunk follows; **entity** chunks carry the data.

| Chunk kind | Payload structure | Example |
|------------|-------------------|---------|
| `status` | `{code: number, message: string}` | `{code: 200, message: "OK"}` |
| `total` | `{value: number}` | `{value: 1500}` |
| `entity` | `object` | `{id: 1, name: "item"}` |
| `entity:{tag}` | `object` (tagged) | `{id: 1, name: "item"}` under `kind: "entity:logs"` |

The `entity:{tag}` form labels content in mixed-type responses.

```json
[
  { "kind": "status", "payload": { "code": 200, "message": "OK" } },
  { "kind": "total", "payload": { "value": 1500 } },
  { "kind": "entity", "payload": { "name": "file1.txt", "size": 1024, "type": "file" } },
  { "kind": "entity", "payload": { "name": "file2.txt", "size": 2048, "type": "file" } }
]
```

In streaming mode each chunk is flushed as it is produced; in buffered mode all chunks are collected and the array is sent at once. Either way the client parses one consistent structure.

Errors use the same format:

```json
[
  { "kind": "status", "payload": { "code": 404, "message": "File not found" } }
]
```

## Streaming vs buffered

| Aspect | Streaming | Buffered |
|--------|-----------|----------|
| Response start | Immediate | After completion |
| Memory usage | Very low (O(1)/chunk) | Full dataset size |
| Best for | Large data, live updates, >100-item lists, long-running ops | Small queries, single-entity reads, simple/mobile calls |

A directory of 10,000 files starts returning results in ~50ms streamed vs ~2000ms buffered, using ~95% less server memory.

## Troubleshooting

| Symptom | Checks |
|---------|--------|
| Streaming seems slow | Stable connection? High server load? Compare with `?stream=false`. |
| Response appears incomplete | Client may not support chunked encoding — test with `curl` or a modern browser; fall back to `?stream=false`. |
| Streaming not working | Confirm the endpoint emits multiple chunks; verify HTTP-library compatibility; test directly with `curl`. |
